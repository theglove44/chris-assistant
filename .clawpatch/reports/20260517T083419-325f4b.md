## low: memory search builds a RegExp from unescaped user input

id: fnd_sig-feat-cli-command-272fa5f4a3-_3f4241ff67
category: bug
confidence: high
triage: confirmed-bug
status: open
feature: Node source src/cli/commands#1 (feat_cli-command_272fa5f4a3)

evidence:
- src/cli/commands/memory.ts:174-177 (registerMemoryCommand)

`chris memory search <query>` first finds matches with a plain lowercase substring check, but then uses the raw query as a regular expression for highlighting. Queries containing regex metacharacters such as `[` or `(` can throw `SyntaxError: Invalid regular expression`, causing the command to fail after it has already found a literal match. Other metacharacters also change highlighting semantics away from the literal search behavior.

recommendation:
Escape the query before constructing the highlighting regex, or avoid regex entirely by doing case-insensitive literal highlighting with index calculations.

test analysis:
No linked tests are included for memory search, especially queries containing regex metacharacters.

suggested regression test:
Add a test for `memory search` with a query like `[` or `a+b` against mocked memory content and assert the command prints highlighted literal matches without throwing.

minimum fix scope:
Escape user input in the highlighting logic in `src/cli/commands/memory.ts`.

repro:
If a memory file contains a literal `[`, run `chris memory search '['`; the substring match can succeed and then `new RegExp('([)', 'gi')` throws.

## high: logs --lines is interpolated into a shell command

id: fnd_sig-feat-cli-command-272fa5f4a3-_f1e7eae9b1
category: security
confidence: high
triage: confirmed-bug
status: open
feature: Node source src/cli/commands#1 (feat_cli-command_272fa5f4a3)

evidence:
- src/cli/commands/logs.ts:30-32 (registerLogsCommand)

The non-follow logs path passes the user-controlled --lines option through a template string to execSync, which invokes a shell. A value such as `--lines '10; touch /tmp/pwned'` would be parsed by the shell as an additional command. The follow path already uses spawn with an argv array, so this is an avoidable command-injection bug at the user-input/process-exec boundary.

recommendation:
Replace execSync with execFileSync or spawn using an argument array, e.g. `execFileSync("npx", ["pm2", "logs", PM2_NAME, "--nostream", "--lines", String(opts.lines)], { stdio: "inherit" })`. Also validate that --lines is a positive integer before passing it to pm2.

test analysis:
No linked tests are included for the CLI logs command or for shell metacharacter handling in command options.

suggested regression test:
Add a CLI unit test or child-process integration test that invokes the logs command with a malicious --lines value and asserts the pm2 invocation receives it as a single argv value or rejects it as invalid, without executing shell metacharacters.

minimum fix scope:
Update `src/cli/commands/logs.ts` non-follow branch to avoid shell execution and validate `opts.lines`.

repro:
Run `chris logs --lines '1; touch /tmp/chris-assistant-injected'` and observe that the shell command after the semicolon can execute.

## low: Synchronous exceptions in PM2 callback leave the CLI promise unsettled

id: fnd_sig-feat-cli-command-5ab4ccecb2-_02235d0248
category: bug
confidence: medium
triage: risk
status: open
feature: Node source src/cli (feat_cli-command_5ab4ccecb2)

evidence:
- src/cli/pm2-helper.ts:18-26 (withPm2)

withPm2 assumes the supplied callback always returns a Promise. If the callback throws synchronously before returning, the exception escapes the pm2.connect callback, pm2.disconnect() is skipped, and the outer Promise is never rejected in the intended control flow. That can leave CLI commands hanging or produce an uncaught exception instead of a normal command failure.

recommendation:
Wrap the callback invocation with Promise.resolve().then(() => fn(pm2)) and put pm2.disconnect() in a finally-style path so both synchronous throws and rejected promises disconnect and reject consistently.

test analysis:
No linked tests were included for src/cli, and the current code has no visible regression coverage for withPm2 error paths.

suggested regression test:
Add a unit test that mocks pm2.connect/disconnect, invokes withPm2 with a synchronously throwing callback, asserts the returned promise rejects, and asserts disconnect was called exactly once.

minimum fix scope:
Change only withPm2 in src/cli/pm2-helper.ts and add focused coverage for synchronous callback failure.

repro:
Call withPm2(() => { throw new Error('boom'); }) from a CLI command or test. The synchronous throw bypasses both the .then and .catch branches where disconnect and reject are handled.
